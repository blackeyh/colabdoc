Architecture and Repository Structure

The final system is intentionally a single-repo full-stack PoC, but the runtime responsibilities are still clearly separated.

Runtime View

Browser Client
React + Tiptap + Yjs
→
FastAPI Backend
REST + WebSocket Relay
DB + AI Provider
SQLite / Postgres-compatible + OpenAI-compatible

The browser owns editor state and collaboration state; the backend owns auth, permissions, persistence, export, and provider-backed AI.

Repository Layout

  • `backend/`: routers, models, AI providers, exporters, WebSocket manager.
  • `frontend/`: React app, Tiptap editor, Yjs logic, sidebar panels, tests.
  • `tests/`: backend pytest suite.
  • root docs: README, deviations, architecture addendum, demo checklist, detailed report.

Why Single Repo

  • Easier for a grader to clone and run locally.
  • Lower demo risk than a more distributed deployment.
  • Still preserves clear frontend/backend responsibility boundaries.
  • Avoids pretending the repo has services or packages that do not actually exist.

Frontend Role

Renders auth screens, dashboard, editor, AI sidebar, permissions, versions, and live presence using React components centered around `EditorPage.jsx` and `EditorTextarea.jsx`.

Backend Role

Provides REST APIs for auth, documents, permissions, versions, AI, plus the authenticated collaboration WebSocket endpoint and static frontend hosting.

Data Role

Stores canonical document JSON, permissions, versions, and AI interactions in the database while collaboration snapshots stay lightweight and in-memory.

Architectural Positioning The final system is simpler than a production multi-service platform, but it is internally consistent. This was a deliberate priority after the earlier report-vs-repo mismatch feedback.

Entity Purpose Why It Matters

Frontend Design and Editor Experience

The frontend is organized so auth, dashboard, editor behavior, and sidebar workflows stay separated but coordinated.

Main Frontend Areas

  • `App.jsx` and `main.jsx` for app bootstrapping and routing.
  • `LoginPage.jsx` and `RegisterPage.jsx` for auth entry points.
  • `Dashboard.jsx` and `NewDocModal.jsx` for document management.
  • `EditorPage.jsx` as the orchestration layer for one document session.
  • `EditorTextarea.jsx` for Tiptap + Yjs + remote cursor integration.
  • Sidebar panels for AI, AI history, permissions, versions, and active users.

API and Session Handling

  • `api.js` attaches the access token to requests.
  • On `401`, it performs a single refresh flow using the refresh token.
  • After a successful refresh, it retries the failed request once.
  • The WebSocket hook follows the same session-recovery idea when a token expires.

Dashboard

Lists accessible documents, supports document creation, and provides the main authenticated entry into the editor workflow.

EditorPage

Coordinates loaded document data, collaboration state, export, AI, versions, permissions, and save behavior for a single open document.

EditorTextarea

Binds Tiptap to Yjs, renders collaborator carets and selections, and keeps AI-applied edits inside normal editor undo/redo history.

Rich-Text Capability

  • Headings
  • Paragraphs
  • Ordered and unordered lists
  • Code blocks
  • Inline emphasis and inline code
  • Undo and redo

Compatibility Layer

Earlier documents may still exist in a simple `{text:"..."}` format.

`contentCompat.js` normalizes both the legacy text-only shape and the final Tiptap JSON shape so older data remains readable instead of requiring a destructive migration.

Frontend Design Principle The UI is split into focused components so AI, versions, permissions, and active-user presence can evolve independently without turning the editor into one monolithic file.

Realtime Collaboration and Conflict Handling

The final collaboration path is one of the biggest improvements over the earlier PoC because connected edits now merge at the CRDT layer.

1
Authenticated Join The client opens `/ws/documents/{id}?token=...`. The backend validates the JWT, checks document access, then emits `init` metadata and the current room state.
2
Yjs Sync Clients exchange `crdt_update` and `crdt_snapshot` messages so concurrent connected edits merge without relying on last-write-wins full-document replacement.
3
Presence Layer Separate cursor and typing messages drive active-user display, collaborator carets, collaborator selection highlights, and typing indicators.
4
Durable Persistence The frontend still sends durable canonical content to the backend so versions, export, and AI use the persisted document rather than only one peer's transient collaborative state.

Important Message Types

Message Purpose
`init`Seeds the room with title, role, active users, and collaboration state.
`crdt_update`Incremental Yjs update from another collaborator.
`crdt_snapshot`Full room snapshot used for late join or reconnect sync.
`sync_request`Requests peers to replay the latest snapshot.
`cursor`Broadcasts selection/caret location.
`typing`Broadcasts activity without changing document content.
`reset`Broadcasts a version restore to connected collaborators.

What Improved Compared With the Earlier Flow

  • Concurrent edits from connected users now merge through Yjs.
  • Presence is visually obvious through in-editor carets and highlights.
  • Late joiners and reconnecting users can bootstrap from a current snapshot.
  • Version restore is synchronized back into the live room.
Authenticated Rooms CRDT Merge Snapshot Replay Remote Carets Typing Presence
Important Trade-Off The backend keeps the latest opaque room snapshot in memory instead of running a dedicated shared Yjs persistence service. This is much stronger than full-document overwrite syncing, but it is still a PoC collaboration backend rather than a horizontally scaled production system.

Section 6

AI Assistant Architecture

The AI flow was upgraded from a blocking pattern into a streamed, audited, role-restricted subsystem that behaves like part of the editor rather than a disconnected demo widget.

AI Backend Flow

  1. User selects text and chooses an action such as rewrite or summarize.
  2. Backend builds a prompt from the action, selected text, and truncated document context.
  3. Provider streams output through `text/event-stream`.
  4. Frontend renders `delta` chunks progressively.
  5. User explicitly accepts, edits and accepts, or rejects the suggestion.
  6. The final decision is recorded in `ai_interactions`.

Event Stream Model

  • `meta`: interaction row created and stream opened.
  • `delta`: partial generated text chunk.
  • `done`: stream completed with final suggestion.
  • `error`: provider failed; partial text is preserved when possible.

Cancel is handled through `AbortController`, and the backend records the interruption state rather than silently dropping the interaction.

Provider Abstraction

`OpenAIProvider` supports any OpenAI-compatible endpoint, while `NullProvider` keeps the full AI UX testable without network access or vendor billing.

Access Control

Only `owner` and `editor` roles can invoke AI. `viewer` and `commenter` are blocked in both the UI and backend route layer.

Audit Trail

The final history stores selected text, generated prompt, provider/model, output, provider status, user action, and final applied text.

AI Safety and Usability Characteristics

Concern Final Handling
Premature auto-editing AI output is never auto-applied. The user must explicitly accept or edit-accept.
Read-only misuse Viewer and commenter roles are blocked in the UI and receive `403` from the backend if they call AI directly.
Long generation feedback Streaming provides partial output instead of a blank waiting period.
Undoability Accepted AI insertions remain part of normal editor history and are covered by `frontend/tests/EditorTextarea.test.jsx`.
Traceability The document-level history panel shows prompt/model/output/status and user resolution.

Alignment With Assignment 1 and Feedback Resolution

The final repo does not try to silently imitate the broader Assignment 1 architecture. It explains what changed, what stayed, and why.

Preserved Product Ideas

  • Collaborative document editing.
  • Role-based sharing.
  • AI-assisted writing workflow.
  • Version history and restore.
  • Export of the current document.
  • Separation between frontend and backend concerns.

Intentional Simplifications

  • Local JWT auth instead of Auth0.
  • No external email or object-storage integrations.
  • Single-repo layout rather than a larger service decomposition.
  • FastAPI relay plus in-memory snapshot cache instead of a dedicated Yjs persistence service.
  • HTML and TXT export instead of binary office-format generation.

Resolution of Earlier Feedback

Earlier Issue Final Resolution
System context diagram omitted the authentication provider or left it ambiguous. Final docs explicitly show backend-managed JWT auth and stop implying an external provider that the runtime does not use.
Long-running AI handling was deferred to synchronous flows. AI is now streamed, cancellable, progressively rendered, logged, and covered by backend/frontend tests.
Repository structure did not match the documented layout. The final docs describe the actual single-repo layout and reviewer flow instead of a larger imagined structure.
Report mismatched delivered auth and realtime stack. `DEVIATIONS.md` and `ARCHITECTURE_ADDENDUM.md` now explain the shipped auth and Yjs/FastAPI collaboration model directly.
README configuration exposed secrets or encouraged unsafe setup. The repo uses `.env.example` as a template and generates a local JWT secret during first-run setup.
Automated tests were missing. The final repo includes backend tests, frontend tests, a submission rehearsal test, and an AI undo proof.
Reading This Section Together With The Repo README, deviations, architecture addendum, demo checklist, and this report are intended to be read together. The goal is to describe the shipped repo directly rather than imply a larger architecture that was not delivered in the final PoC.
